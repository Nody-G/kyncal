# Kyncal — Context Memory

## Architecture
- **Stack**: Next.js 16 (App Router), TypeScript, Tailwind CSS v4, shadcn/ui (base-ui)
- **Stockage**: localStorage côté client (pas de backend)
- **Déploiement**: Vercel
- **Thème**: Dark par défaut, mode clair disponible
- **Exports**: jsPDF (PDF), xlsx (Excel), CSV natif

## RÈGLES MÉTIER DU PLANNING (IMPORTANT)

### Règle 1 : 1 rôle maximum par cascadeur par jour
- Un cascadeur ne peut être assigné qu'à **un seul rôle par jour**
- Implémenté via `assignesCeJour` (Set<string>) dans la boucle des jours
- Si un cascadeur est déjà assigné ce jour, il est exclu des candidats pour les rôles suivants

### Règle 2 : Jours de repos obligatoires (typeRepos)
- Chaque cascadeur a un `typeRepos`: "6j/1" (6 jours travail → 1 jour repos) ou "5j/1"
- Quand `joursDepuisDernierRepos >= typeRepos`, le cascadeur **doit** être en repos
- Le repos est **prioritaire** sur toute assignation de rôle
- Le repos se cumule : si un cascadeur a 0 jour de repos programmé et dépasse le seuil, il prend 1 jour de repos
- Si le cascadeur a des jours de repos programmés dans la semaine, le compteur se réinitialise

### Règle 3 : Jours consécutifs max (via typeRepos)
- Le max de jours consécutifs est déterminé par `typeRepos`: "6j/1" → 6 jours max, "5j/1" → 5 jours max
- Quand `joursDepuisDernierRepos >= maxJours` (calculé par `getMaxJoursAvantRepos`), le cascadeur est forcé en repos
- Il n'y a PAS de `minJoursConsecutifs` / `maxJoursConsecutifs` global — c'est le typeRepos qui contrôle
- Les `ContrainteEnchainement` (par rôle) ont `joursMin` et `joursMax` pour contraindre l'enchaînement dans un rôle spécifique

### Règle 4 : Équilibrage des jours travaillés entre cascadeurs
- L'algorithme calcule la **moyenne** des jours travaillés parmi tous les cascadeurs actifs
- Pénalité forte (écart × 500 points) pour les cascadeurs au-dessus de la moyenne
- Cette pénalité dépasse la continuité (200) pour **forcer l'équilibrage**
- Résultat : les cascadeurs avec beaucoup de rôles ne monopolisent pas les jours de travail
- Le compteur `totalJoursTravailles` est incrémenté à chaque assignation

### Règle 5 : Spectacles actifs uniquement
- Seuls les spectacles marqués `actif: true` sont planifiés
- Les spectacles inactifs sont ignorés par l'algorithme

### Règle 6 : Cascadeurs actifs uniquement
- Seuls les cascadeurs marqués `actif: true` sont planifiés
- Les cascadeurs inactifs sont ignorés par l'algorithme

### Règle 7 : Absences respectées
- Les absences déclarées empêchent toute assignation ces jours-là
- Le cascadeur est marqué "absent" dans le planning

### Règle 8 : Jours de diffusion des spectacles
- Chaque spectacle a `joursDiffusion: JourSemaine[]`
- L'algorithme ne planifie un spectacle que sur ses jours de diffusion
- Si un spectacle n'a pas de jours de diffusion, il n'est jamais planifié

### Règle 9 : Rôles et priorités
- Chaque cascadeur a des `roles: RoleCascadeur[]` (spectacleId, roleId, priorité)
- Priorité : "primaire" ou "secondaire"
- L'algorithme favorise les cascadeurs avec le rôle primaire
- Un cascadeur sans rôle pour un spectacle donné ne peut pas y être assigné

### Règle 10 : Score d'éligibilité
- Le score détermine quel cascadeur est choisi parmi les candidats
- Plus le score est **bas**, meilleur est le candidat
- Composantes du score :
  - **Rôle assigné** : primaire = 0, secondaire = 100, sans rôle = Infinity (exclu)
  - **Jours dans le rôle actuel** : bonus si < maxJoursConsecutifs (favorise la continuité)
  - **Jours depuis dernier repos** : pénalité si proche du seuil typeRepos
  - **Équilibrage** : pénalité (écart × 500) si au-dessus de la moyenne des jours travaillés
- `Infinity` = cascadeur exclu (pas de rôle, pas actif, absent, etc.)

## Notes techniques
- shadcn/ui utilise `@base-ui/react` (pas radix) — pas de `asChild`, les Select ont `string | null` dans onValueChange
- Les contraintes d'enchaînement sont configurables via `config.contraintesEnchainement`
- Le recalcul partiel est un placeholder (régénère tout pour l'instant)
- Jours de diffusion: chaque spectacle a `joursDiffusion: JourSemaine[]` — l'algorithme ne planifie que ces jours
- jsPDF: import dynamique `import("jspdf").then((m) => new m.default(...))`

## Données
- Cascadeur: nom, prénom, typeRepos (6j/1 ou 5j/1), roles[], absences[], actif
- Spectacle: nom, description, couleur, roles[], joursDiffusion[], actif
- RoleSpectacle: id, nom, nbCascadeursRequis
- RoleCascadeur: spectacleId, roleId, priorite (primaire/secondaire)
- Saison: nom, dateDebut, dateFin, spectacleIds[]
- Planning: saisonId, entrees[], dateGeneration

## État interne de l'algorithme (CascadeurState)
- `joursTravaillesConsecutifs`: jours travaillés d'affilée (reset à 0 sur repos)
- `joursReposConsecutifs`: jours de repos d'affilée (reset à 0 sur travail)
- `joursDepuisDernierRepos`: jours depuis le dernier repos (reset à 0 sur repos) — C'EST CE CHAMP qui contrôle le repos obligatoire
- `joursDansRoleActuel`: jours consécutifs dans le rôle actuel (reset si rôle change)
- `totalJoursTravailles`: total jours travaillés sur la saison (pour équilibrage)
- `dernierRoleSpectacleId`: ID du dernier spectacle assigné
- `dernierRoleId`: ID du dernier rôle assigné
- `reposPrisCycle`: repos pris dans le cycle actuel (non utilisé dans la logique actuelle)

## Contraintes d'enchaînement par rôle (ContrainteEnchainement)
- Configurées dans `config.contraintesEnchainement`
- Chaque contrainte lie un `spectacleId` + `roleId` à un `joursMin` et `joursMax`
- `respecteEnchainement()` vérifie que le cascadeur a atteint le min avant de quitter un rôle
- `respecteEnchainement()` vérifie que le cascadeur n'a pas dépassé le max dans un rôle
